----------------------------------------------------------------------
1. PROJECT GOAL
----------------------------------------------------------------------

This project is a legal-domain RAG assistant.

Main purpose:
- User asks a legal question in the UI
- System retrieves relevant legal content from stored documents
- System generates an answer
- System shows the source document and page numbers
- User can click the source and open the PDF directly at the cited page
- System can optionally use an LLM judge to review the answer

The project is designed mainly around:
- Indian legal / statutory / legal-book PDFs
- Qdrant vector retrieval
- Ollama local LLM generation
- Gemini-based feedback/review
- Optional vectorless pageIndex retrieval
- Fine-tuning and evaluation support

4. CURRENT INFERENCE MODELS
----------------------------------------------------------------------

Current model split:

- Main answer generation:
  qwen2.5:7b

- Judge / reviewer:
  gemini-2.5-flash

- Query rewriter:
  gemini-2.5-flash

Meaning:
- Answer is generated locally with Ollama + Qwen
- Review is done using Gemini
- Query rewrite after failed review is also done using Gemini

5. HOW DOCUMENT INGESTION WORKS
----------------------------------------------------------------------

When preparing the knowledge base, the project reads PDFs from the data folder.

Source folder:
- data/*.pdf

Main steps during indexing:

1. Read each PDF
2. Extract text page by page
3. Clean whitespace
4. Split content into chunks
5. Preserve metadata:
   - source file
   - page start
   - page end
   - chunk index
6. Build vector embeddings using Ollama embeddings model
7. Store vectors in Qdrant
8. Store text chunks in a SQLite FTS5 pageIndex backing index

6. RETRIEVAL MODES
----------------------------------------------------------------------

The system supports 2 retrieval modes:

1. VECTOR
- Uses embeddings
- Query is embedded
- Qdrant similarity search returns top chunks

2. PAGEINDEX
- Uses keyword/full-text matching only
- No embeddings are needed
- Uses SQLite FTS5

Why this matters:
- Vector retrieval helps semantic matching
- pageIndex retrieval helps exact term matching and page-based traceability


----------------------------------------------------------------------
7. VECTORLESS RAG
----------------------------------------------------------------------

Vectorless RAG means the system can retrieve context without using embeddings.

Implemented using:
- lexical_index.py
- build_lexical_index.py
- SQLite FTS5

What happens:
- User query is matched against indexed chunk text
- Matching chunks are returned
- Context is built from those chunks
- Qwen generates the answer from that context

Why it is useful:
- cheaper than embedding-based retrieval
- useful when semantic embedding infrastructure is not available
- good for exact legal terms, sections, names, and keywords

Limitations:
- weaker than vector retrieval for paraphrased questions
- may miss semantically relevant but differently worded content


----------------------------------------------------------------------
8. CONTEXT BUILDING
----------------------------------------------------------------------

After retrieval, the backend formats the retrieved chunks into context blocks.

Each block contains:
- source file name
- page range
- score
- chunk text

The answer model receives:
- user question
- context blocks

The model is instructed to:
- answer clearly
- stay grounded in context
- cite the context blocks like [1], [2]
- respond in the same language as the user if possible


----------------------------------------------------------------------
9. FEEDBACK / REVIEW LOOP
----------------------------------------------------------------------

This is one of the key advanced features of the project.

What it does:
- Does not trust the first generated answer blindly
- Uses a second model to judge whether the answer is good enough

Judge inputs:
- original user question
- retrieved context
- draft answer

Judge outputs:
- verdict
- relevance_score
- groundedness_score
- completeness_score
- rationale

If answer is weak:
- Gemini rewrites the retrieval query
- system retrieves again
- system generates another answer
- judge checks again

This continues for a limited number of rounds.

Why this matters for presentation:
- It makes the system more reliable than plain single-pass RAG
- It shows an "LLM evaluating another LLM" design
- It is inspired by recursive/self-correcting RAG ideas

13. FINE-TUNING PURPOSE
----------------------------------------------------------------------

Why fine-tuning is added:
- To make the model behave more like a legal assistant
- To improve legal-domain language style
- To potentially improve answer structure and user-facing quality

Important distinction:
- RAG gives grounded document knowledge
- Fine-tuning gives domain behavior/style

So fine-tuning should complement RAG, not replace RAG.


----------------------------------------------------------------------
14. FINE-TUNING PIPELINE
----------------------------------------------------------------------

Fine-tuning folder:
- fine_tuning/

Files:
- prepare_sft_dataset.py
- train_lora.py
- evaluate_models.py
- plot_finetune_results.py

What happens:

STEP 1: PREPARE DATASET
- Load local JSONL or Hugging Face dataset
- Parse chat turns or QA format
- Convert to:
  - prompt
  - completion
- Split into:
  - train_sft.jsonl
  - eval_sft.jsonl

Supported chat sources:
- Hugging Face conversation datasets
- Gemma-style turn format
- INST-style turn format

STEP 2: TRAIN MODEL
- Load base model from Hugging Face
- Quantize model to 4-bit
- Attach LoRA adapters
- Train only the adapters

This is QLoRA-style training because:
- base model is quantized to 4-bit
- LoRA adapters are trained on top

STEP 3: SAVE ADAPTER
- Adapter is stored in output folder
- Base model is not fully rewritten

----------------------------------------------------------------------
15. BASE MODEL CHOICE FOR FINE-TUNING
----------------------------------------------------------------------

Since project inference uses Qwen, fine-tuning should ideally stay in Qwen family.

Good choices:
- Qwen/Qwen2.5-3B-Instruct
- Qwen/Qwen2.5-7B-Instruct

Why:
- keeps training family aligned with inference family
- reduces mismatch between tuned behavior and production behavior

Why not randomly switch to Gemma:
- even if a dataset is Gemma-style formatted, the base model should match project goals
- family consistency matters more than dataset token marker style


----------------------------------------------------------------------
16. WHAT HAPPENS AFTER FINE-TUNING
----------------------------------------------------------------------

After training:
- you get a trained adapter
- you do NOT automatically get a production-ready Ollama model

Meaning:
- training succeeds
- adapter is saved
- but there is still a bridge needed to serve/use the tuned model as an actual candidate model

So there are 3 conceptual stages:

1. Prepare training data
2. Train adapter
3. Serve or merge the tuned model for evaluation/inference

21. HONEST LIMITATIONS
----------------------------------------------------------------------

These are important to mention in a presentation honestly.

1. PDF deep-linking is page-level, not sentence-level
2. Judge loop improves reliability but does not guarantee perfect answers
3. Lexical retrieval may miss semantic matches
4. Fine-tuning currently saves an adapter; serving/merging is still a separate deployment step
5. Final fine-tuning improvement plots require:
   - baseline model access
   - candidate tuned model access
   - evaluation dataset
   - successful evaluation run
